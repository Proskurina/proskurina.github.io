Day 23-24: Bugs days

24 September 2015

It all started with Apple iPhone Software Update (to version 9.0). iPhone just stuck and did not want to move anywhere from the page 'slide to upgrade'.

Bugs that we make and catch during our learning process are not so epic of course, but sometimes an extra letter, a typo, lack of a dot or comma can easily ruin several hours of your life.

And now I got a flue bug, had to stay at home and decided to write this post.

So, how to find a bug?

For me, as a beginner, the easiest way to catch a bug is to print out everything that's going on in my code. I put a print (p or puts) statement before and after every line of my code, with a short explanation of what I am printing. Then I literally can read when and where a bug appears.

But there are better ways. For example, using pry or byebug.

Pry is very similar to irb, but has some extra features.
To use pry, first you need to install it using gem install pry or include gem 'pry' to your Gemfile and run bundle install.

Then you need to require pry in a file, where you want to find a bug, and print binding.pry before the place where you think the bug can be. When you run your program, it will stop at binding.pry point. And then you can print next lines in pry, receiving feedback on what is going on after each line.

Using byebug is similar. You install it using gem install byebug or include it into your Gemfile and run bundle install. Then you require byebug in a file anf print byebug in your file where you want your program to stop running. And then you can play with byebug, the same way as you play with irb or pry.

Bugs have no chances!

Yana Proskurina
